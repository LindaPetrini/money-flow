# Plan 10-02: Fix INFRA-04 — First-Run Detection Decoupled from accounts.length

## Goal

Fix the UX defect where a returning Chrome user who has cleared their data (or freshly installed Chrome) sees the first-run onboarding card instead of the reconnect banner. The root cause is `isFirstRun = needsFsaPrompt && accounts.length === 0` — the `accounts.length === 0` condition is true for both genuine first-run users AND returning users who happen to have no accounts.

## Context

**Bug location:** `src/App.tsx` line 21:
```typescript
const isFirstRun = needsFsaPrompt && accounts.length === 0;
```

**Problem:** After `seedIfEmpty()` runs in `main.tsx`, accounts are always seeded with defaults on first run. But if a returning user clears their browser data (or is on a fresh Chrome install with an existing folder that got wiped), `accounts.length === 0` becomes true again — yet the user is NOT on a genuine first run, they just have no data.

**Correct signal for first run:** Whether the settings have been initialized/seeded. After `seedIfEmpty()` in `main.tsx`, the settings store always has `overflowRatios.length > 0` if it was seeded. An uninitialized settings state (fresh first-run before `seedIfEmpty` seeds it) will have `overflowRatios.length === 0`.

**However**, looking at the flow more carefully:

1. `main.tsx` calls `loadSettings()`, then `seedIfEmpty()` which seeds `overflowRatios` — so after init, a first-run user WILL have `overflowRatios.length > 0`.
2. A returning user with existing folder: `loadSettings()` loads their stored settings — also has `overflowRatios`.
3. A returning user with cleared data: `loadSettings()` returns the default (empty `overflowRatios`), then `seedIfEmpty()` seeds them — so they also end up with `overflowRatios`.

The fundamental issue: `accounts.length === 0` is NOT a reliable proxy for "is this genuinely the first time this browser has run the app". The accounts always get seeded by `seedIfEmpty()`.

**Wait — re-reading `seedIfEmpty()`:** It seeds accounts if `accounts.length === 0`. So BOTH a genuine first-run user AND a returning user whose data was cleared will have accounts seeded to DEFAULT_ACCOUNTS. In this case `accounts.length` after init will be `> 0` for both scenarios.

**The real scenario for the bug:** A user on Chrome who:
1. Has NOT yet chosen a folder (needsFsaPrompt = true)
2. Has accounts in IDB fallback (or no accounts — but wait, seedIfEmpty runs before render)

Since `seedIfEmpty()` always runs before `createRoot()`, by the time `App` renders, `accounts.length` is always > 0 (unless storage is IDB and accounts were explicitly deleted by user outside the app).

**Simpler correct fix:** The first-run card should only show when `needsFsaPrompt && !settingsInitialized` — but settings is always initialized before render.

**Actual simplest correct fix:** Use `needsFsaPrompt` alone as the first-run trigger, but distinguish "has existing data" from "no data" differently. The correct distinction is:

- `needsFsaPrompt && accounts.length === 0` — WRONG (accounts always > 0 after seedIfEmpty)
- `needsFsaPrompt` alone — shows onboarding to ALL returning Chrome users, wrong
- `needsFsaPrompt && settings.overflowRatios.length === 0` — only true if seedIfEmpty hasn't run yet, which never happens

**The real correct fix (from CONTEXT.md):** The simplest correct approach is `isFirstRun = needsFsaPrompt` ONLY — show the "Choose data folder" card whenever FSA prompt is needed, whether first-run or returning user. The reconnect banner (`needsFsaPrompt && !isFirstRun`) would then never show. But CONTEXT.md says the reconnect banner IS a valid UI element.

**Final decision (per CONTEXT.md):** The CONTEXT.md says: "Planner should check what the Phase 7 isFirstRun logic actually does and pick the least-invasive correct fix." And: "Alternative simpler approach: `isFirstRun = needsFsaPrompt` only (show onboarding whenever FSA prompt is needed, regardless of accounts)".

Looking at the current App.tsx: the reconnect banner shows when `needsFsaPrompt && !isFirstRun`. If `isFirstRun = needsFsaPrompt`, then `!isFirstRun` when `needsFsaPrompt` is true is always false — meaning reconnect banner never shows. That's fine: both the first-run card and the reconnect banner prompt the user to `handleGrantAccess()`. The UI difference is cosmetic.

**The least-invasive correct fix:** Change line 21 from:
```typescript
const isFirstRun = needsFsaPrompt && accounts.length === 0;
```
to:
```typescript
const isFirstRun = needsFsaPrompt;
```

This means: whenever Chrome needs FSA permission, show the full onboarding card (which has the "Choose data folder" button). The reconnect banner is redundant when the full card is shown, so suppressing it is correct behavior.

**State Decision Record:** This was already decided in STATE.md (Phase 07-hardening 07-02 entry):
> "isFirstRun = needsFsaPrompt && accounts.length === 0 distinguishes genuine first-run from returning user with lapsed permission"

This decision was *incorrect* and is the bug INFRA-04 describes. The fix supersedes that decision.
