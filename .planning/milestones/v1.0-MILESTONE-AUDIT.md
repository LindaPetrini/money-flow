## Integration Issues

### Medium — ALLOC-02: Floor Item Coverage Over-Marking

**File:** `src/features/invoice/InvoicePage.tsx` lines 74–103

When `Done` is clicked, `InvoicePage` marks floor items as covered by matching `destinationAccountId` from the generated moves. If two floor items share the same destination account and only the higher-priority one is funded (invoice exhausted mid-stabilize), both items get marked covered — because the match is by account ID, not floor item ID.

The `AllocationMove` type lacks a `floorItemId` field, so the engine doesn't communicate which specific floor item each move belongs to.

**Impact:** Default configuration has one floor item (`floor-rent`) pointing to `acc-everyday`, so the bug does not trigger on the happy path. It activates when a user creates two floor items targeting the same account — a valid configuration.

**Fix:** Add `floorItemId?: string` to `AllocationMove` in `src/types/domain.ts`; populate it in `allocationEngine.ts` stabilize(); use it in `InvoicePage.handleDone()` instead of `destinationAccountId` for coverage matching.

### Low-UX — INFRA-04: First-Run Onboarding Card Not Shown in Chrome

**File:** `src/main.tsx` lines 16–21, `src/App.tsx` line 21

The `isFirstRun = needsFsaPrompt && accounts.length === 0` check in App.tsx always evaluates to `false` on first Chrome load because `seedIfEmpty()` runs before `createRoot()` and populates `accountStore` with 5 default accounts. By render time, `accounts.length === 5`, not 0. Users see the reconnect banner instead of the "Welcome to Money Flow" card.

The app is still functional — the reconnect banner provides the "grant access" action — but the designed onboarding experience is never delivered.

**Fix:** Track first-run state separately (e.g. `localStorage['has_run_before']` set after successful FSA grant), rather than inferring it from account count. Or move the `isFirstRun` check before `seedIfEmpty()` in `main.tsx`.
